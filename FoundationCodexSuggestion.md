# Foundation Codex Suggestion

## Core Thesis

The product should not be positioned as "AI makes mockups." That space already has strong pressure from tools like Figma Make, v0, Uizard, Bolt, Lovable, and design-to-code platforms.

The sharper wedge is:

> Mockups as living, structured product context: generated by AI, refined by humans, consumed by AI agents.

This is the product.

The core problem is not that teams need another visual mockup tool. The deeper problem is that product intent, design rationale, requirement traceability, and implementation context leak across handoffs.

Today, teams move through:

```text
PRD -> Design -> Comments -> Handoff -> Code
```

At each step, context is lost. AI coding agents make this worse because they often receive visual context without the decision context behind it.

The opportunity is to create an AI-native mockup and handoff workspace where the mockup is not just a picture of the product. It is a structured agreement about what should be built.

## Reframing The Product

The original single-file `.mockup.html` idea remains valuable, but it should become an export format, not the main source of truth.

The source of truth should be a cloud-hosted mockup document with:

- PRD requirements
- user flows
- screens
- components
- design rationale
- annotations
- acceptance criteria
- implementation hints
- design tokens
- version history
- unresolved questions
- AI-readable structured context

Think of the product as:

```text
PRD + Design System + Human Decisions + Mockups + Agent Context
= Living Product Spec
```

The UI mockup is the visible surface. The structured context graph is the real asset.

## Positioning

Do not position this as a Figma replacement.

Position it as:

> An AI-native mockup and handoff workspace where product teams turn PRDs into implementation-ready, traceable UI specs.

Or more simply:

> Mockups that explain themselves to humans and AI agents.

The buyer pain is:

- AI-generated screens lose product intent.
- PRDs do not survive handoff.
- Figma files do not explain requirement traceability.
- Developers and AI agents receive visual context but not decision context.
- PM, design, and engineering keep re-answering the same questions.
- Generated code misses design-system and product constraints.

The product should make the mockup the shared brain between PM, design, engineering, and AI agents.

## Product Pillars

### 1. Generate

Users upload or connect a PRD, design guide, screenshots, existing app, or Figma file. The system generates flows and mockups, not just isolated screens.

### 2. Trace

Every screen and component maps back to requirements, assumptions, acceptance criteria, and design rationale.

### 3. Collaborate

PMs, designers, developers, and AI agents all comment on the same artifact. Comments are structured, resolvable, role-aware, and part of the document history.

### 4. Constrain

The product learns the team's design system, code components, product vocabulary, UX rules, and implementation patterns. This is where quality improves beyond generic AI mockups.

### 5. Handoff

The system exports to HTML, Figma, GitHub issues, Linear tickets, Markdown specs, or an MCP/API endpoint for Codex, Claude Code, Cursor, and other AI coding agents.

AI agents need structured design and product context, not just screenshots.

## Architecture Shift

Instead of:

```text
/mockup PRD + guide -> one HTML file
```

Move to:

```text
Sources -> Product Context Graph -> Generated Mockup -> Human Review -> Agent Handoff
```

The server product should have these core objects:

- Workspace
- Project
- Source Document
- Requirement
- Flow
- Screen
- Component
- Design Token
- Annotation
- Decision
- Version
- Export
- Agent Context

Every visual node should have provenance:

```json
{
  "component": "search-bar",
  "screen": "search-results",
  "requirementIds": ["REQ-42"],
  "rationale": "Primary search is the dominant user action.",
  "acceptanceCriteria": ["Debounced input", "Keyboard navigable"],
  "designSystemMapping": "SearchInput",
  "implementationHint": "Use existing app/components/SearchInput.tsx"
}
```

This provenance layer is the magic, not the pixels.

## MVP

The first sellable version should be narrow and excellent:

1. Web app with projects and shareable mockup links.
2. Upload PRD and design guide.
3. Generate a multi-screen interactive mockup.
4. Side panel showing requirement traceability per element.
5. Human annotations with roles: PM, Designer, Developer.
6. Version history and visual diff.
7. Export as `.mockup.html`.
8. Export as Markdown implementation spec.
9. Agent endpoint: copy context for Cursor, Codex, Claude, or expose an MCP server.

Do not start with full Figma-level editing. Start with structured edits:

- Change this screen to support bulk actions.
- Mark this requirement as P0.
- Add empty, loading, and error states.
- Map this button to our existing Button component.
- Generate developer handoff.

The editor should feel more like an AI product workbench than a canvas tool.

## Moat

The moat is not AI generation. Generation will be commoditized.

The moat is accumulated structured context:

- company design system
- preferred flows
- product terminology
- reusable requirement patterns
- accepted decisions
- codebase component mappings
- implementation feedback
- what changed between mockup and shipped code

Over time, the product should know:

> For this company, a settings page uses these components, these states, these permission rules, these empty-state conventions, and these implementation patterns.

That kind of organizational product memory is hard for generic AI mockup tools to replicate.

## What Changes From The Original Foundation

Soften or remove these assumptions:

- One file as the primary product principle.
- No server.
- No dependencies.
- The browser file is the collaboration layer.
- Mockup targets one viewport.
- Designer edits directly inside the file.

Keep these principles:

- Context is data.
- Human and AI readable.
- Portable export.
- Annotations are first-class.
- AI-native handoff.
- Meet users where they are.

Add one central principle:

> The mockup is not a picture of the product. It is a structured agreement about what should be built.

## Go-To-Market

Start with teams where handoff pain is expensive:

- B2B SaaS teams
- internal tools teams
- agencies building client portals
- product squads using AI coding agents
- startups where PM, design, and dev roles blur

Avoid selling first to pure visual designers. They will compare the product to Figma.

Sell to product and engineering teams trying to make AI-assisted development reliable.

The first paid promise:

> Turn PRDs into implementation-ready mockups with requirement traceability and AI-agent handoff.

Possible pricing:

- Individual / Pro: mockup generation and exports
- Team: shared projects, comments, design systems, GitHub and Linear integrations
- Enterprise: SSO, private models or VPC, audit logs, custom component mapping

## Strong Recommendation

Do not build AI Figma.

Build the missing context layer between product thinking and AI implementation.

The artifact should still render beautifully. But the business value is that a human can review it, a PM can defend it, a designer can refine it, a developer can implement it, and an AI agent can understand it without guessing.
