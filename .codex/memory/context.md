Tvastr is being productized from a no-server `/mockup` skill into an AI-native mockup/spec handoff product. The core product thesis is: mockups built with AI, for AI and humans in the loop. The durable positioning is not “another design tool” or “AI Figma”; it is a living, structured mockup/spec artifact that preserves product intent, requirement traceability, design rationale, annotations, states, acceptance criteria, and implementation constraints for both humans and coding agents.

The original `.mockup.html` concept remains important as a portable export/artifact, but the product direction now allows a server and shareable hosted links. The preferred mental model is: created locally, reviewed visually, edited by humans, consumed by agents.

Important constraints and UX decisions: PMs/builders should not be forced to upload PRDs into a new web portal as the primary workflow. PRDs and design notes often stay local or in existing docs. Tvastr should meet users where they already work: Codex, Claude Code, Cursor, Copilot, terminal, local docs, GitHub repos, and optionally Figma. A website cannot directly read local files; local files should be read by a local runner such as a CLI, local MCP server/tool, or editor extension, with user permission. The cloud product can generate, store, and publish the resulting mockup/spec.

Recommended first creation workflow: `npx tvastr mockup ./prd.md --design ./design.md --publish`. The local CLI reads files, sends content to the backend, writes a local `feature.mockup.html`, and optionally returns a hosted link such as `https://tvastr.app/m/abc123`. Agent-native workflow: a local MCP/tool exposes something like `tvastr.create_mockup` so Codex/Claude Code/Cursor can create a mockup from local paths and receive a shareable link.

The product should have three main surfaces: 1) local generator via CLI/MCP/agent tool, 2) shareable browser-based mockup editor for humans, and 3) agent-readable endpoint/API/MCP for coding agents. The browser editor should be simple and clear for designers; all manual edits can live there. Designers may optionally bring their own API key for AI-assisted edits, and may import/export to Figma or use paper/pencil if they choose. Designers should not need Codex, Claude Code, Cursor, or a paid AI subscription to participate.

Stakeholder workflow: PM/builder keeps PRD/design notes local, uses an existing AI tool or CLI to generate a mockup, reviews product intent, edits requirement labels/priorities/copy/acceptance criteria/comments, and shares the link. Designer opens the link, manually edits layout/copy/spacing/colors/fields/hierarchy/states, adds design notes and missing states, optionally uses AI with their own API key or imports/exports to Figma, then marks design ready. Developer receives the link, reviews flow and structured context, adds feasibility/component/API notes, gives the link to Codex/Claude Code/Cursor/Copilot, and links implementation PRs back to the mockup. Coding agents should consume structured context, not just screenshots/HTML: requirement IDs, screen flow, components, states, annotations, design notes, acceptance criteria, implementation hints, and component mappings.

Validated product pain cluster: teams do not need another design canvas; they need AI-generated mockups that preserve product intent, requirement traceability, design rationale, and implementation constraints all the way into code. The market evidence discussed included Figma MCP/Code Connect, OpenAI+Figma, and design handoff/requirements traceability sources, supporting the thesis that AI agents need structured design/product context and component mappings, not just pixels.

Offline browser editing should be designed as an embedded edit mode inside each `.mockup.html` artifact, not a separate local editor app. The end result of this feature should be a self-editing mockup artifact: a designer opens one HTML file in a browser, enters edit mode, visually changes the mockup, validates it, and leaves with a new valid `.mockup.html` file containing the UI, flow behavior, embedded `tvastr-context`, traceability, audit history, and sync metadata.

For offline edit mode, designers should feel like they are editing the mockup rather than maintaining HTML. They should be able to open/review the actual mockup offline, enter edit mode without account/network/build step, visually select meaningful UI elements, edit copy, adjust layout and style, manage screens and states, edit simple interactions, add annotations/notes/decisions, validate the artifact, and save or export the updated portable HTML artifact.

Variants should be first-class and accessible in embedded edit mode, including default, loading, empty, error, disabled, permission denied, OTP, 2FA, locked account, success, and presentation/color variants. However, the editor should show only one active view at a time, modeled as `Flow -> Screen -> State/Variant -> Viewport`, rather than rendering all variants side by side by default.

The recommended MVP implementation for variants is materialized variants with a single active renderer: each important screen/state variant exists as readable HTML in the artifact, while inactive variants are hidden or stored as templates. This is preferred over patch-based variant rendering for MVP because it is easier for designers, agents, validation, and local editing to inspect and modify.

The embedded artifact should maintain a variant registry in `tvastr-context`, including screens, variant IDs/types/names, and the current `activeView` such as screen, variant, and viewport. A variant switcher should let designers move among screens, content/system/access/auth/presentation variants, and responsive viewports while keeping only the selected combination visible.

Offline edit mode should make edit scope explicit. Designers need to know whether an edit applies only to the current variant, all variants of a screen, related states, or all components using a design token. When editing the active variant, the editor should update both the visible DOM and embedded `tvastr-context`; changes to one variant should not accidentally affect others unless the designer chooses a broader scope such as apply-to-all.

The saved/exported artifact should remain one `.mockup.html` file containing all screens, all variants, the current active view, audit log, context mappings, and sync metadata. Reopening the file should show the last or intended active view directly, while preserving all other variants for switching and future sync/validation.
